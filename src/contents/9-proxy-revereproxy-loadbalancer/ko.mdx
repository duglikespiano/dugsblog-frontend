웹 시스템을 구성하거나 운영할 때, **proxy**, **reverse proxy**, **load balancer**에 대해 종종 듣게 됩니다.

그들은 매우 비슷해 보이고, 실제로 겹치는 분야도 있습니다. 하지만 그들은 각각 다른 문제를 해결하며 네트워크 구성에서 각자 다른 위치에서 작용합니다.

## 간단 비교

| Feature           | Proxy                   | Reverse Proxy              | Load Balancer         |
| ----------------- | ----------------------- | -------------------------- | --------------------- |
| Who it represents | Client                  | Server                     | Server group          |
| Position          | Client side             | Server side                | Server side           |
| Main purpose      | Control outbound access | Protect & optimize servers | Distribute traffic    |
| Hides             | Client identity         | Server details             | Backend servers       |
| Typical use case  | Company network, VPN    | Web servers, APIs          | High-traffic services |

## Proxy

Proxy는 클라이언트와 인터넷 사이에서 작용합니다.

클라이언트가 인터넷에 직접적으로 접근하는 대신 proxy를 먼저 통하게 됩니다.

### 이런 문제를 해결합니다

- 클라이언트의 실제 IP를 숨깁니다
- Outbound access를 제한하고 필터링합니다
- Bandwidth 사용량을 줄이기 위해 response를 캐싱합니다

```txt
Client ➡️ Proxy ➡️ Internet
```

## Reverse Proxy

Reverse proxy는 서버들 앞에 위치하며 인터넷을 향합니다.

Client는 서버와 직접 소통한다고 생각합니다만, 사실은 reverse proxy와 소통하게 됩니다.

### 이런 문제를 해결합니다

- 서버 내부 구조를 숨깁니다
- TLS (HTTPS) 조작을 중앙화합니다
- Responses를 캐싱합니다
- 보안규칙을 적용합니다 (접속량 제한, IP 차단)

```txt
Client ➡️ Reverse Proxy ➡️ Server
```

## Load Balancer

Load balancer는 트래픽들을 여러 서버에 걸쳐 분배합니다.

이로 인해 한 서버에 과부하가 걸리는 것을 방지합니다.

### 이런 문제를 해결합니다

- 트래픽량을 핸들링합니다
- 가용성을 개선합니다
- Horizontal scaling을 가능하게 합니다

```txt
Client
  ➡️ Load Balancer
    ➡️ Server A
    ➡️ Server B
    ➡️ Server C
```
